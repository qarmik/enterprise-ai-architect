Core Principles
The non-negotiable rules that govern all 42 months. These are not motivational guidelines. They are structural constraints that protect the long-term effort from short-term variability.
§ 1.1
1.1 — Study Capacity and Effort Architecture
§ 1.1.1
The Planning Unit
7–10 hours per week is the planning unit. Morning sessions of 60–90 minutes are the primary build window. Evenings are bonus only — never scheduled, never counted toward the weekly target. When a surge week is available — two consecutive days of high energy and professional quiet — up to 12 hours may be used. No obligation to sustain a surge. § 1.1.2
Done Definition Before Every Session
Before opening any resource, write one sentence defining what done looks like for this session. "Done = I have committed one function that reads the CSV without crashing." When done arrives, stop. No extension. No guilt. The session boundary protects the next session's energy as much as it protects the current one.Deepseek

Artifacts Over Modules
A week of reading with no committed artifact is a lost week. A week of one working thing with no reading is a successful week. The presence of a committed, tested artifact is the only valid measure of a phase's progress. No amount of notes, videos, or tutorial completion substitutes.

§ 2
2 — Three-Tier Energy System
Energy levels vary unpredictably for anyone managing a demanding professional career alongside serious technical study. Attempting a full session under reduced energy produces failure, which produces guilt, which compounds fatigue into a multi-day loss. The three-tier system interrupts this before it begins.Deepseek
🟢
Green Mode
Full scheduled session. Build, debug, commit. All learning mechanisms active. This is the default you plan for. Phases advance on Green sessions.
🟡
Yellow Mode
Read-only session. Read documentation, read your own code, watch one specific video. No building, no debugging. Approximately 50% cognitive load. Maintains continuity without risking the failure-guilt cycle.

On the first Yellow mode use: write in your study log: "I chose Yellow today. That was correct engineering."
🔴
Red Mode
10-minute minimum only. Open your laptop. Read your last commit message. Write one study log line. Close. The pathway stays open. The streak survives.

On community days in Red mode: read one thread in a relevant Discord. Do not post. Write one sentence in the study log about what you read.Deepseek
§ 3
3 — Operating Protocols
§ 3.1
Getting Unstuck — 7 Steps
Read the error message bottom to top. Name the problem in one sentence. Search precisely: tool name + exact error + operating system. Read the top three results in priority order. Try one fix at a time. After 45 minutes of no progress, ask for help. After solving: write the error, the fix, and why it happened in the error log — same day.

Asking for help is not failure. The 45-minute limit prevents the false belief that persistence through confusion is the same as learning.
§ 3.2
Multi-Month Disruption Re-Entry
If away for more than 3 weeks — Session 1: read last 3 commit messages and session-state.md only. Write one study log line. Stop. Session 2: rebuild the last artifact from scratch without looking at your own code. Session 3: review the Sunday task list, write 3 new specific tasks. Session 4 onward: resume forward progress. Do not catch up. Do not extend sessions to compensate. Just continue.ChatGPT
§ 3.3
Quarterly Review Protocol
Every 3 months: a 30-minute calendar appointment. Four written questions: (1) How interested am I in enterprise AI architecture right now, 1–10? (2) What has changed in my life since last quarter that affects this plan? (3) Am I on track — if not, do I adjust timeline or goal? (4) What would need to be true for me to legitimately quit this path?

Question 4 is the most important. Knowing the exit criteria prevents both drift and sunk-cost continuation. Persistence into the wrong goal is not virtue.

Exit criteria — all four must be simultaneously true to legitimately quit: interest below 4 for three consecutive quarters; concrete alternative path identified; learning documented and repurposable; discussion with a trusted person completed.DeepseekQwen
§ 3.4
AI Usage Policy — Phase-Gated
Phase 0: No AI assistance on any code. Build everything yourself.
Phase 1+: AI as debugging assistant only — after 45 minutes of independent effort. Structured prompt required: paste error + your attempted fix + your hypothesis. AI may explain errors. It may not generate new code. Close the AI. Write the code from memory. If you cannot write it from memory, the AI did the learning.Deepseek

§ 6.1
6.1 — LLM Serving Mechanics
§ 6.1.1
The Inference Pipeline — From Request to Token
§ 6.1.1.1
Prefill and Decode Phases
Prefill: process the entire input prompt in parallel to populate the KV cache — compute-bound. Decode: generate output tokens sequentially, attending to all prior context — memory-bandwidth-bound. This distinction drives every batching and hardware selection decision you will make for the next four phases.
§ 6.1.1.2
KV Cache and PagedAttention
The KV cache stores key-value pairs computed during prefill. Without paging, it causes memory fragmentation — large contiguous allocations that cannot be reused across requests. PagedAttention uses virtual memory paging to allocate KV cache in non-contiguous blocks, dramatically improving GPU memory utilisation. Read the original PagedAttention blog post before writing any vLLM code.
§ 6.1.1.3
Continuous Batching and Chunked Prefill
Static batching waits for a fixed batch size — unacceptable for variable-length enterprise workloads. Continuous batching inserts new requests into the decode step without waiting for current requests to complete. Chunked prefill splits long prefill sequences to reduce GPU memory spikes. These two features make vLLM production-viable for high-throughput serving.
§ 6.1.2
Quantization and Precision Trade-offs
§ 6.1.2.1
Precision Formats
FP32: 4 bytes per parameter. BF16/FP16: 2 bytes, standard inference. INT8: 1 byte, ~2× memory reduction. INT4/GPTQ/AWQ: 0.5 bytes, ~4× reduction, measurable accuracy degradation on complex reasoning. FP8: emerging hardware standard. For regulated-industry deployment: any quantization applied to a model used in high-stakes contexts must be benchmarked for accuracy degradation before deployment — precision loss is an audit-relevant system characteristic.
§ 6.1.2.2
VRAM Budget Arithmetic
Model weights (parameters × bytes/parameter at chosen precision) + KV cache (2 × layers × heads × head_dim × sequence_length × batch_size × bytes) + activation overhead (~10–15% of weights) = total VRAM required. Compute this before any GPU is provisioned. It is a planning gate, not a post-hoc discovery.
§ 6.1.3 — Operational Math Thread · Phase 1
The probability pipeline of token generation: Raw logits from the model's final linear layer (one value per vocabulary token). Softmax: exp(logit_i) / Σexp(logit_j) — converts logits to a probability distribution summing to 1. Temperature scaling: divide logits by T before softmax. T→0 makes the distribution sharper (deterministic); T→∞ makes it uniform (random). Top-p sampling: sort tokens by probability, keep the smallest set whose cumulative probability exceeds p.

This is not abstract. It is the exact parameter space you tune when a language model produces hallucinated text in a high-stakes application — the parameters that control the model's tendency to generate plausible-sounding but incorrect output.

§ 7.1
7.1 — API Integration and Tool Systems
§ 7.1.1
OpenAI-Compatible API Patterns
Messages array (system / user / assistant / tool roles). Temperature, top_p, max_tokens, stop sequences. Streaming responses. The same API call works against your vLLM endpoint by changing only the base_url and model parameters — this is the enterprise deployment pattern for regulated environments.
§ 7.1.2
Agent Orchestration with LangGraph
§ 7.1.2.1
State Machines and Cyclic Graph Architecture
LangGraph models agent behaviour as a directed graph with state. Nodes are processing steps (LLM calls, tool calls, human review gates). Edges are conditional transitions. State persists across the full execution cycle. Cyclic graphs — loops, retries, replanning — are necessary for robust enterprise agents. Linear chains break on the first unexpected tool response. ADR-006: framework selection, with alternatives considered.
§ 7.1.2.2
Checkpointing and State Serialisation
Agent state checkpointed to persistent storage. When a long-running agent is interrupted — by a system failure, a human review gate, or a policy check — it resumes from the last checkpoint. This is the technical basis for the HITL systems built in Phase 4.
§ 7.1.2.3
Framework Abstraction Discipline
LangGraph is the chosen framework for Phases 2–4. This choice is documented in ADR-006. The curriculum does not teach LangChain, CrewAI, AutoGen, or Haystack simultaneously — these frameworks share conceptual patterns but change APIs frequently. When LangGraph is superseded, ADR-006 is updated, not the curriculum.

§ 7.2
7.2 — Vector Retrieval and RAG Systems
§ 7.2.1
Embedding Systems and Chunking
An embedding model converts text into a dense vector — semantically similar text produces vectors with high cosine similarity. Use a self-hosted sentence-transformers model to avoid external API dependency. Chunking strategies: fixed-size (simple, lossy at boundaries), sentence-aware, recursive character splitting, semantic chunking. For documents with dense regulatory or technical content, paragraph-level chunking with overlap preserves context that fixed-size chunking loses.
§ 7.2.2
Vector Databases and Similarity Search
FAISS: in-memory, Python-native, ideal for development and benchmarking. ChromaDB: persistent with metadata filtering. Both support cosine and dot product similarity. Production vector database selection (Qdrant, Weaviate, pgvector) documented in ADR-008. Retrieval evaluation: Precision@k, Recall@k, MRR computed on a held-out evaluation set before any RAG system is considered ready.
§ 7.2.3 — Operational Math Thread · Phase 2
Vector similarity mathematics: Cosine similarity = (A·B) / (|A|×|B|). Why normalise vectors before computing dot products. Euclidean distance = √(Σ(Aᵢ-Bᵢ)²). When cosine similarity outperforms Euclidean distance for text (almost always for normalised embeddings — magnitude is meaningless). The curse of dimensionality: why similarity search degrades in very high dimensions and why HNSW (Hierarchical Navigable Small World) graphs are used instead of brute-force search.

8.1 — GraphRAG and Knowledge Graph Retrieval
§ 8.1.1
Why Vector RAG Fails on Complex Domain Queries
Vector RAG retrieves semantically similar chunks. It fails when the answer requires reasoning across multiple documents that are not individually similar to the query. A question requiring information from three different regulatory documents — each relevant to a different aspect of the answer — will not be correctly retrieved by flat vector search. A knowledge graph makes these cross-document relationships explicit and traversable.
§ 8.1.2
Entity Extraction and Graph Construction
GraphRAG uses an LLM to extract entities (organisations, regulations, dates, concepts) and relationships (governs, references, supersedes, applies_to) from documents. These are stored in a graph database. Complex queries traverse graph paths rather than doing vector similarity search. NetworkX for development (in-memory, Python-native). Neo4j for persistence. ADR-009: graph database selection.
§ 8.2
8.2 — Evaluation Pipelines
§ 8.2.1
LLM-as-Judge Evaluation
Faithfulness: does the answer contain only information supported by the retrieved context? Relevancy: does the answer address the question? Correctness: is the answer factually accurate against ground truth? These three metrics (DeepEval implementation) run against every response in the evaluation set. A regression in any metric triggers a CI/CD pipeline failure. ADR-010: evaluation framework selection.
§ 8.2.2
CI/CD Integration
Every pull request to main triggers: unit tests → integration tests → evaluation suite on 30 standardised prompts → benchmark regression check (throughput must not degrade more than 10% vs baseline). Evaluation failures block merge. This is the production deployment standard that separates a robust system from a prototype.
§ 8.2.3 — Operational Math Thread · Phase 3
Evaluation statistics: Why a hallucination rate of 2.1% on 50 samples is statistically indistinguishable from 0%. Sample size requirements for detecting a 1% change in faithfulness score. Inter-rater reliability (Cohen's Kappa) when using multiple LLM judges. An evaluation suite of 10 prompts is not statistically meaningful for enterprise deployment decisions — this is the mathematical justification for why evaluation set sizing matters.
§ 8.3
8.3 — Observability Stack
§ 8.3.1
OpenTelemetry for AI Systems
OpenTelemetry's GenAI Semantic Conventions define standard attribute names for LLM spans: gen_ai.request.model, gen_ai.usage.input_tokens, gen_ai.usage.output_tokens. Using these standard attributes means your traces are readable by any OTel-compatible backend without custom parsing. Distributed tracing across the full agent chain: API gateway → orchestrator → retrieval → generation → validation. Each hop is a trace span. Without distributed tracing, you cannot identify whether a slow response is caused by retrieval, LLM, or validation.ChatGPTKimi
§ 8.3.2
Prometheus, Grafana, and SLO Definition
Counter: total requests, errors, tokens consumed. Gauge: active requests, VRAM utilisation, queue depth. Histogram: request latency with p50/p95/p99 buckets. Grafana dashboards persist these metrics visually. SLO definition: availability target, p99 TTFT, error budget — all documented. An SLO that is not documented is not an SLO.Kimi

§ 9.1
9.1 — Human-in-the-Loop System Architecture
§ 9.1.1
HITL Design Patterns
§ 9.1.1.1
Asynchronous HITL
When an agent must pause for human review, it serialises its complete execution state — the graph node it is at, all prior messages, all tool results, the pending action — to durable storage. The review queue displays this state in a structured, human-readable format. On approval, the state is deserialised and execution resumes from the exact paused point. Asynchronous HITL is required for any agent action with financial, legal, or regulatory consequences.
§ 9.1.1.2
Risk Scoring and Escalation Chains
Not every action requires the same level of review. Define a risk scoring model: read-only data retrieval → no review. Report generation → automated validation gate. External communication draft → manager review. Sensitive action → compliance officer review plus second sign-off. Risk thresholds are documented and enforced programmatically by the agent router. This maps directly to the approval matrix and maker-checker concepts from regulated-industry operations — the translation is literal, not metaphorical.
§ 9.1.1.3
Rollback Architecture
Every agent action that modifies external state must be reversible. Design for rollback before designing for action. Any simulated transaction executed by an agent has a corresponding cancel operation. Rollback mechanisms are tested as explicitly as forward actions.
§ 9.2
9.2 — Compliance Architecture
§ 9.2.1
AI Bill of Materials (AIBOM)
An AIBOM documents every component of an AI system for regulatory audit: base model (name, version, training data provenance, licence), fine-tuned adapters, quantization parameters, dependencies (framework versions, inference server version), deployment configuration. Generated automatically from the repository and versions.md. Committed with each release. Can you answer: who created this model, on what data, under what licence, and what changes have you made? If not, the model cannot be deployed in a regulated environment.
§ 9.2.2
NIST AI Risk Management Framework
NIST AI RMF organises AI risk management around four functions: GOVERN (policies and accountability), MAP (risk identification), MEASURE (risk quantification), MANAGE (risk response). For each major portfolio artifact, produce a one-page NIST AI RMF mapping. This is the document format that enterprise risk and compliance functions speak. Building this mapping ability is worth more in regulated-industry AI roles than any certification.
§ 9.2.3
ISO 42001 Alignment
ISO 42001 is the AI Management System standard (published 2023). It is a certifiable standard — unlike NIST AI RMF which is voluntary. European and Asian regulated entities deploying AI post-EU AI Act increasingly require ISO 42001 alignment. Document how the Phase 4 governance system satisfies the key clauses: 6.1 (risk identification), 8.4 (AI system impact assessment), 9.1 (monitoring and measurement).

10.2 — Advanced Infrastructure: Triton — Conceptual Fluency Only
Scope Constraint — Explicitly Enforced
This section teaches conceptual fluency and 1–2 small Triton artifacts. Not compiler engineering. The objective: (a) read a Triton kernel and understand what hardware operation it implements, (b) explain why a custom kernel outperforms PyTorch on a specific pattern, (c) intelligently direct infrastructure teams who specialise in kernel optimisation. This scope serves the architect role. Expanding it produces a kernel engineer specialisation that is not the target.
§ 10.2.1
SPMD Execution Model and Fused Operations
Triton programs execute with Single Program Multiple Data semantics: the same program runs on many data points simultaneously, indexed by a block grid. A fused kernel combines multiple operations into a single GPU kernel launch, eliminating intermediate memory reads/writes between separate operations. FlashAttention is fast because it fuses attention score computation and softmax into a single pass over the attention matrix, staying within SRAM rather than round-tripping through HBM.
