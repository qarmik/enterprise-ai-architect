Chapter One
What a Computer Is Actually Doing
And why a banker already understands it better than they think

Every computer, regardless of how sophisticated it appears, does exactly one thing: it follows instructions. Not intelligent instructions. Not thoughtful instructions. Precise, sequential, unambiguous instructions written by a human being and stored in a file. The computer has no judgment. It has no intuition. It has no concept of what those instructions mean. It simply executes them, one after another, at extraordinary speed.

This is the most important thing to understand before writing a single line of code. The machine is not smart. You are. Your job is to write instructions precise enough that a machine with no judgment can follow them without error.

The SBI Analogy
Think of a branch office that processes loan applications. The counter staff do not decide whether a loan is approved. They follow a procedure manual: check document checklist, verify KYC, enter data into CBS, submit for sanctioning. The procedure manual is the program. The counter staff is the processor. CBS is the operating system.
Program = the procedure manual · Processor = the staff executing it · Operating system = CBS · Your Python code = a new chapter added to the procedure manual

The Four Layers Every Computer Has

Every computer — from your ThinkPad to the GPU server that will run vLLM in Phase 1 — has four layers. Understanding these layers now prevents confusion for the next 42 months.

⚙️
Layer 1 — Hardware
The physical machine
What it is
The physical components: CPU, RAM, storage, GPU. Metal, silicon, electricity.
What it does
Performs actual calculations. Stores data. Moves electricity through circuits.
SBI analogy
The physical branch building, vaults, teller counters, computers. The infrastructure that makes operations possible.
Why it matters
In Phase 1, you will care deeply about one specific piece of hardware: the GPU's VRAM (Video RAM). The size of VRAM determines which AI models you can run.
🏛️
Layer 2 — Operating System
The manager between hardware and software
What it is
Linux, macOS, or Windows. Software that manages the hardware and lets other programs use it.
What it does
Manages memory, files, processes, network connections. When your Python program says "save this file," the OS decides exactly where on disk it goes.
SBI analogy
CBS — the core banking system that manages all operations and allocates resources. Every branch function goes through CBS. Every program goes through the OS.
Your environment
GitHub Codespaces runs Ubuntu Linux. You are learning on Linux. All enterprise AI servers run Linux. This is not incidental.
🐍
Layer 3 — The Runtime
Python: the interpreter
What it is
Python is a program that reads your Python code and translates it into instructions the OS and hardware understand.
What it does
You write human-readable code. Python converts it to machine operations line by line as it runs. This is why it is called an "interpreted" language.
SBI analogy
A Finacle module that takes branch officer input (human-readable forms) and translates it into CBS transactions (machine operations). You fill a form; Finacle handles the rest.
📄
Layer 4 — Your Code
The instructions you write
What it is
Text files with a .py extension. Plain text that Python can read and execute.
What it does
Defines exactly what the computer should do in a given situation. Every artifact you build in this curriculum is a collection of these files.
The key insight
Your code does not run on the hardware directly. It goes through Python → OS → Hardware. Each layer translates for the layer below.

What Actually Happens When You Run python3 hello.py

This is the sequence. Read it slowly. Every step is real.

1
You type python3 hello.py in the terminal and press Enter. You are telling the OS: "Find the program called python3 and give it the file hello.py to work with."
2
The OS finds Python. It looks in a list of known locations for a program called python3. Finds it. Loads it into RAM.
3
Python opens hello.py. It reads the file as plain text. Every character you typed is now in Python's memory.
4
Python reads line 1. Interprets it. Tells the OS what operation to perform. The OS tells the hardware. Hardware executes. Result comes back up the chain.
5
Python reads line 2, 3, 4... This continues until the last line. This entire process happens in milliseconds.
6
Python exits. OS reclaims the memory. The terminal returns control to you.
Why This Matters for the Entire Curriculum
Every error you will ever encounter comes from one of these layers failing to understand what the layer above it asked for. A SyntaxError means Python could not read your code (Layer 3 failed). A FileNotFoundError means the OS could not find a file you referenced (Layer 2 failed). An ImportError means Python could not locate a library (Layer 3 searching Layer 2 storage). Knowing which layer failed tells you where to look for the fix.

Chapter Two
The Terminal
Your CBS for the machine — the interface where real work happens

The terminal is not a relic from the past that proper engineers have moved beyond. It is the primary interface for every production system you will build in this curriculum. Every server, every deployment, every debugging session happens in the terminal.

When you open GitHub Codespaces, you see two things: a file editor on the left and a terminal at the bottom. The file editor is for reading and writing code. The terminal is for executing it. This distinction matters because the terminal is where the machine actually responds to you in real time.

The CBS Analogy
Imagine two ways to interact with CBS. One: a graphical interface with menus, buttons, and forms — designed for branch staff who need hand-holding. Two: a command-line interface used by CBS administrators who type exact commands and get exact responses. The terminal is the command-line interface for your entire computer. It is more powerful than the GUI because it is more precise.
Terminal command = CBS administrator command · Output = CBS response · Error message = CBS rejection reason · Working directory = current CBS module open

The Vocabulary of the Terminal

prompt
The line ending with $ where you type. It shows your username, machine name, and current location. @qarmik ➜ /workspaces/enterprise-ai-architect (main) $ — this tells you exactly who you are, where you are, and which Git branch you are on.
command
An instruction you type at the prompt. ls, cd, python3, git. The terminal sends it to the OS for execution.
argument
Additional information you give to a command. python3 hello.py — here hello.py is the argument. It tells python3 which file to run.
flag
An option that modifies a command's behaviour. Usually starts with - or --. ls -la — -la tells ls to show all files including hidden ones, in long format.
working directory
The folder your terminal is currently "inside." When you run a command, the OS looks for files relative to this location. pwd shows where you are.
path
The address of a file or folder. /workspaces/enterprise-ai-architect/phase-0/hello.py is a full path. phase-0/hello.py is a relative path — relative to where you currently are.
output
What the command prints to the terminal after running. Sometimes it is information. Sometimes it is an error. Sometimes it is nothing — silence means success for many commands.
exit code
Every command finishes with a number. 0 means success. Anything else means failure. Your CI pipeline checks exit codes to know if tests passed.

The Pipeline Pattern — Learn Once, Use Forever

The pipeline is one of Linux's most powerful ideas. It lets you chain commands together so the output of one becomes the input of the next. The symbol is | — called "pipe."

The Pipe Analogy
Imagine a physical processing line in an SBI back office. Desk 1 extracts all transaction slips from a box and passes them forward. Desk 2 takes only the FAILED ones. Desk 3 counts how many failed by each error code. Desk 4 sorts that count. Desk 5 shows the top 10. The pipe | is the conveyor belt between desks.
# The complete pipeline pattern — read left to right like a conveyor belt
cat error.log        # Desk 1: open and output the full log file
  | grep "ERROR"     # Desk 2: keep only lines containing "ERROR"
  | sort             # Desk 3: sort them alphabetically
  | uniq -c         # Desk 4: count how many times each unique error appears
  | sort -rn        # Desk 5: sort by count, largest first
  | head -10        # Desk 6: show only the top 10

You will use this exact pattern on vLLM logs in Phase 1, NGINX access logs in Phase 2, and OpenTelemetry trace outputs in Phase 3. Learn it once now. It never changes.

Chapter Three
Git: The Transaction Ledger of Code
Why version control is not optional and how it works at the level you need to know

Git is not a backup system. It is a permanent, tamper-evident, chronological record of every change ever made to your codebase. It is the most important engineering habit in this curriculum. It is also something you already understand intuitively — because you work in a bank.

In banking, every transaction is recorded with an amount, a timestamp, a description, and an operator ID. You cannot delete a transaction from the ledger. You can only add a correction entry. The ledger is the truth. Disputes are resolved by reading the ledger. This is exactly how Git works.

The CBS Transaction Ledger Analogy
Every CBS transaction has: a transaction ID (unique), a timestamp (when), an operator ID (who), a description (what), and the before/after state (the change). A Git commit has the exact same structure: a commit hash (unique ID), a timestamp, an author, a commit message (description), and a diff (the before/after change). In CBS, you cannot alter a posted transaction. In Git, you cannot alter a committed commit without the history showing the alteration.
git commit = posting a transaction · commit hash = transaction ID · commit message = transaction narration · git log = CBS transaction history · git diff = before/after comparison report

The Three States of a File in Git

Every file in your repository is always in one of three states. Understanding this eliminates 80% of Git confusion.

📝
State 1 — Modified (Working Directory)
Changed but not yet staged
What it means
You edited the file. Git knows it changed. But you have not told Git you want to include this change in the next commit.
SBI analogy
You filled in a transaction slip but have not submitted it to the system yet. It sits on your desk.
How to see it
git status — shows modified files in red.
📋
State 2 — Staged (Index)
Marked for the next commit
What it means
You have told Git "include this change in the next commit." The change is queued but not yet permanent.
SBI analogy
The transaction slip has been submitted to the checker for verification. It is in the queue. Not yet posted.
How to get here
git add filename — moves the file from modified to staged.
🔒
State 3 — Committed (Repository)
Permanently recorded in the ledger
What it means
The change is now in Git's permanent history. It has a unique hash. It cannot be erased without the erasure itself becoming part of the history.
SBI analogy
The transaction is posted in CBS. It appears in the ledger. It is now a permanent record. Any correction requires a new entry.
How to get here
git commit -m "description" — moves all staged changes into a permanent commit.

The Five Git Commands That Run Your Entire Workflow

You will use exactly five Git commands for every session in this curriculum. They always run in this order.

The Five Commands — Always in This Order
git status
What it does: Shows which files have changed, which are staged, which are untracked. Analogy: asking "what transactions are pending, what are in the queue, what are new entries?" before doing anything. Run this first. Always. Before you add, before you commit.
git add .
What it does: Stages ALL changed and new files. The . means "everything in the current directory." Analogy: submitting all pending transaction slips to the checker queue at once. Be careful — this includes files you may not want. Use git add filename to stage specific files.
git commit -m "phase-0: add hello.py"
What it does: Creates a permanent commit with the message you provide. -m means "message follows." Analogy: posting all queued transactions with a batch narration. The message is the narration in the ledger — it must be specific and meaningful. "Updated files" is as useless as "Misc transactions" in a CBS narration.
git push origin main
What it does: Sends your commits to GitHub. origin is the name Git gives to your GitHub repository. main is the branch name. Analogy: uploading the branch's day-end data to the central server for consolidation. Until you push, your commits exist only on your local machine.
git log --oneline -5
What it does: Shows the last 5 commits as one line each. Hash + message. Analogy: reading the last 5 entries in the operations register at the start of a shift. This is how every session starts — reading the ledger to know where you are.

Commit Message Discipline

The commit message is your narration in the ledger. A CBS entry saying "Transaction" is useless. A commit message saying "updated files" is equally useless. The format in this curriculum is:

# Format: prefix: present-tense description under 72 characters
phase-0: add transaction_log_analyser with tests
fix: remove accidental content from requirements.txt
docs: write ADR-002 for file output decision
session: update state after phase-0 comprehension week

# Never write these:
updated files        # What files? What update?
fix stuff            # What stuff? What fix?
WIP                  # This is a permanent record, not a sticky note
asdf                 # This is vandalism of your own ledger
The Most Common Git Error
The shell stuck at a > prompt after git commit -m. This means your quote was not closed. You opened " but never closed it. The shell is waiting for the closing quote. Fix: press Ctrl+C to cancel. Re-run the command with a properly closed quote on one line.

What .gitignore Is and Why It Exists

Some files should never be committed to your repository. Generated output files that change every run. Secret credentials. The venv/ folder containing installed packages (100MB+ that can be recreated from requirements.txt). The .gitignore file tells Git to ignore these.

Your .gitignore — What Each Line Means
venv/
Ignore the virtual environment folder. This folder contains hundreds of installed packages. It is recreatable from requirements.txt. Never commit it — 100MB of files with zero informational value to anyone reading your code.
phase-0/reports/
Ignore generated reports. Your transaction_log_analyser generates timestamped .txt files here every run. These are output, not source. Like not filing printed receipts in the procedure manual.
phase-0/data/
Ignore fetched data. Your API client saves CSV files here. Same principle — generated output, not source code.
__pycache__/
Ignore Python's compiled cache files. Python creates .pyc files as a performance optimisation. They are regenerated automatically. Never commit them.
.env
Ignore environment variable files. This will matter in Phase 1 when you have API keys. NEVER commit API keys or secrets. Once committed and pushed, they are in the public internet permanently even if you delete them later.

Chapter Six
What Python Is and Why It Exists
Before writing a function, understand why functions exist at all

Python is a human-readable language for telling a computer what to do. It was designed to be readable — more like English prose than machine notation. This choice was deliberate: code is read more often than it is written, and readable code is maintainable code.

Python was created in 1991 by Guido van Rossum. He wanted a language that was simple enough that a programmer could express ideas without fighting the language's syntax. His explicit design goal was: "There should be one obvious way to do it." Python is now the dominant language in AI, data science, and infrastructure automation — not because it is the fastest language, but because it communicates intent clearly.

The Procedure Manual Analogy
A well-written SBI procedure manual uses plain language. "Step 1: Verify the customer's KYC documents. Step 2: Enter the account number into CBS. Step 3: If the account status is Dormant, escalate to the Branch Manager." Python code for the same procedure would look nearly identical in structure — steps, conditions, actions. The difference is that Python must be absolutely unambiguous, because the machine cannot ask for clarification.

The Python Vocabulary You Need Before Chapter 7

script
A Python file that runs from top to bottom, doing a task. Your hello.py is a script. It runs, does something, exits.
module
A Python file that contains reusable functions and is imported by other files. When you write import csv, Python loads the csv module.
library / package
A collection of modules. requests is a library. pytest is a library. You install them with pip. They extend what Python can do.
interpreter
The Python program itself — the one that reads your code and executes it line by line. When you type python3, you are starting the interpreter.
syntax
The grammar rules Python enforces. Missing a colon, wrong indentation, unclosed parenthesis — these are syntax errors. Python cannot execute the file until they are fixed.
runtime error
An error that occurs while the program is running — not in the syntax, but in the logic. Trying to open a file that does not exist. Dividing by zero. These only appear when that line executes.
indentation
Python uses 4 spaces to show what belongs inside what. Unlike most languages that use {}, Python uses whitespace. Inconsistent indentation is a syntax error. This is why you never mix tabs and spaces.

Chapter Seven
Variables: The Account Ledger Entry
How Python stores and retrieves information

A variable is a named container that holds a value. When Python encounters a variable, it looks up the name in its memory and retrieves whatever is stored there. This is the most fundamental operation in all of programming — store something, recall it by name.

The Ledger Entry Analogy
In a CBS account ledger, each entry has a field name (Account Number, Customer Name, Balance, Status) and a value (123456789, Rohit Kumar, 45000.00, Active). A Python variable works identically: a name on the left, a value on the right, an equals sign connecting them. The name is how you recall the value later.
Field name = variable name · Field value = variable value · = is assignment, not equality · Looking up a field by name = using the variable name in code

The Four Basic Types You Will Use in Phase 0

Python Data Types — What Each One Is
# str — string: text in quotes account_id = "ACC001" status = "SUCCESS" branch = "Meerut Main"
str = string = text. Anything in single or double quotes is a string. "42" is a string. 42 is not. "SUCCESS" is a string Python can compare to another string. Analogy: any text field in CBS — account holder name, transaction narration, branch code.
# int — integer: whole number total_transactions = 150 failed_count = 12 processing_time_ms = 142
int = integer = whole number, no decimal. You can do arithmetic on ints. failed_count + 1 gives 13. Analogy: count of transactions, number of accounts, days to maturity — any whole number field.
# float — floating point: decimal number amount = 15000.50 failure_rate = 8.0 interest_rate = 7.25
float = floating point = number with decimals. CBS amounts, interest rates, percentages. Analogy: any monetary amount in CBS — always stored with decimal precision.
# bool — boolean: True or False only is_active = True requires_escalation = False file_exists = True
bool = boolean = only True or False. Note the capital T and F — Python is strict about this. Used for yes/no decisions. Analogy: account Active/Inactive flag, KYC verified Y/N, loan approved Y/N.

The Critical Distinction: = vs ==

The Most Common Python Beginner Error
= assigns a value. == compares two values.

status = "SUCCESS" — stores the string "SUCCESS" in the variable called status.
status == "SUCCESS" — asks the question: "is status equal to SUCCESS?" Returns True or False.

Writing if status = "SUCCESS": is a syntax error. You used assignment inside a condition. Python will refuse to run. You must use if status == "SUCCESS":. This error appears in every beginner's code at least once. Now you know before it happens to you.

Function Anatomy — Read Every Symbol

compute_failure_rate — the name. Must describe what it does. Verb + noun. Underscores between words.
(total: int, failed: int) — parameters in parentheses. Two inputs: total (an integer) and failed (an integer). The : int is a type hint — tells the reader what type is expected.

Errors and Exceptions
The exception handling framework — how Python signals and recovers from failure

Errors are not failures. They are Python's way of telling you precisely what went wrong and where. The difference between a beginner and an experienced programmer is not that experienced programmers make fewer errors. It is that they read error messages instead of panicking at them.

The CBS Exception Handling Analogy
When a CBS transaction fails, the system does not silently continue or crash. It generates an exception code (Error 4201: Account not found) with a description, marks the transaction as failed, and routes it to the exception queue for resolution. Python's error system works identically. Each error has a type (FileNotFoundError), a description ("No such file or directory"), and a traceback showing exactly which line caused it.

The Five Error Types You Will Encounter in Phase 0

SyntaxError
Python cannot read your code because the grammar is wrong. Missing colon, unclosed parenthesis, wrong indentation. Python refuses to run the file at all. Fix: read the line number in the error message and look for the syntax violation.
NameError
You used a name that does not exist. NameError: name 'months' is not defined — you typed motnhs somewhere above. Python cannot find months because you never created it by that exact spelling. Case sensitive.
FileNotFoundError
The file path you gave does not exist. FileNotFoundError: No such file or directory: 'phase-0/sample.transactions.csv' — the file is sample_transactions.csv (underscore, not dot). The path must be exact.
KeyError
You tried to access a dictionary key that does not exist. KeyError: 'Status' — you used capital S, but the key is 'status' (lowercase). Dictionaries are case-sensitive.
ValueError
A function received a value it cannot work with. ValueError: No samples provided — this is one you wrote yourself in compute_percentiles. You deliberately raised it when the input list was empty.

Reading a Traceback — Bottom to Top

When Python shows an error, it shows a traceback — the chain of function calls that led to the error. Always read it from the bottom upward. The bottom is the actual error. The lines above it show how you got there.

Chapter Twenty
Why GPU Matters for AI Inference
The physics of AI computation — understood before Phase 1 begins

A GPU is not a faster CPU. It is a fundamentally different kind of processor. Understanding why this matters for AI is the foundation of every Phase 1 decision — what model fits, what precision to use, what throughput to expect.

A CPU — the processor in your ThinkPad or MacBook — is designed for sequential, complex tasks. A few very powerful cores (4–16 typically) that can each handle intricate decision trees, run operating systems, execute business logic. Fast at one thing at a time.

A GPU was designed for graphics: rendering millions of pixels simultaneously. Each pixel calculation is simple (multiply some numbers, add some colours) but there are millions of them. So a GPU has thousands of simple cores — less powerful individually, but enormously powerful in parallel.

The Teller Window Analogy
A CPU is like a branch with 8 highly trained senior officers. Each can handle any complex transaction — loans, forex, escalations, disputes. But there are only 8 windows open at once. A GPU is like a branch with 4,096 data entry operators, each capable of doing one specific simple operation very fast. Running an AI model requires doing the same simple multiplication billions of times. The 4,096 operators finish it while the 8 senior officers are still reading the form.
CPU cores = senior officers (few, powerful) · GPU cores = data entry operators (thousands, simple) · AI matrix multiplication = millions of identical simple operations · VRAM = the operators' working desk space

VRAM — The Most Important Constraint in Phase 1

VRAM is Video RAM — the memory on the GPU card itself. This is distinct from your computer's main RAM. When you run an AI model on a GPU, the model's weights must fit entirely in VRAM. If they do not fit, the model cannot run on that GPU. This is the primary constraint you will manage throughout Phases 1–5.

The Single Most Important Fact in Phase 0
A 7B parameter model (7 billion parameters) in FP16 precision requires approximately 14GB of VRAM.

The arithmetic: 7 × 10⁹ parameters × 2 bytes per parameter (FP16) = 14 × 10⁹ bytes = 14GB.

Do this calculation in your head. Before you provision any GPU in Phase 1. Before you choose a quantization level. This is the planning gate for every infrastructure decision in this curriculum.

The Precision Formats — What They Mean

FP32
32-bit floating point. 4 bytes per parameter. Used for training. High precision. Rarely used for inference — too much memory. A 7B model in FP32 requires 28GB VRAM.
FP16 / BF16
16-bit floating point. 2 bytes per parameter. Standard for inference. 7B model = 14GB. This is what Colab Pro T4 (16GB) can run with minimal KV cache.
INT8
8-bit integer. 1 byte per parameter. 2× memory reduction vs FP16. Some accuracy degradation. 7B model = 7GB. Fits more easily, runs faster.
INT4 / GPTQ / AWQ
4-bit integer. 0.5 bytes per parameter. 4× reduction vs FP16. Measurable accuracy degradation on complex reasoning. 7B model = ~4GB. Used when VRAM is tight.
For Regulated Industry Deployment
Any quantization applied to a model used in high-stakes institutional contexts must be benchmarked for accuracy degradation before deployment. In Phase 1, you will compare FP16 vs INT4 outputs on banking-domain prompts and document the difference in ADR-004. Precision loss is an audit-relevant system characteristic — it must be documented, not assumed acceptable.

Chapter Twenty-Two
Memory Bandwidth vs FLOPS
Why the bottleneck in AI inference is not computation but data movement

FLOPS — Floating Point Operations Per Second — measures how many mathematical operations a GPU can perform per second. It sounds like the right measure of a GPU's AI capability. It is not. For LLM inference, the binding constraint is almost always memory bandwidth — how fast data can move between VRAM and the compute cores.

The Factory and Conveyor Belt Analogy
Imagine a factory that processes raw materials. The workers (GPU compute cores) are extremely fast — they can process 1000 units per second. But the conveyor belt bringing raw materials from the warehouse (VRAM to compute cores) can only deliver 100 units per second. The workers are idle most of the time, waiting for materials. Adding more workers does not help — the bottleneck is the conveyor belt, not the workforce. This is LLM inference: the arithmetic is trivial; reading the model weights from VRAM is the bottleneck.
GPU compute cores = workers · VRAM = warehouse · Memory bandwidth = conveyor belt speed · FLOPS = worker speed · Bottleneck = conveyor belt, not workers

This is why quantization helps throughput even when VRAM is not the constraint: smaller weights (INT4 vs FP16) means the conveyor belt moves the same information in less time. INT4 model weights are 4× smaller — the conveyor belt delivers them 4× faster — and inference throughput improves significantly.

Chapter Twenty-Three
Linear Algebra Intuition
The mathematics of AI — operational understanding, no proofs

This chapter does not teach you to perform matrix calculations. It teaches you what matrices and vectors represent intuitively, because these terms appear constantly in the Phase 1 documentation for vLLM, PagedAttention, and quantization. You need to understand what they mean without needing to do the arithmetic.

Vectors — Positions in Space

The Credit Score Analogy
Imagine describing a loan applicant using 5 numbers: credit score, years of employment, existing EMI burden, account balance, loan-to-income ratio. That 5-number description is a vector in 5-dimensional "applicant space." Two applicants with similar vectors (similar numbers across all dimensions) are similar applicants. This is exactly how AI models represent words and concepts — as vectors in high-dimensional space. Words with similar meanings end up with similar vectors.
Vector = a list of numbers describing something · Dimensions = the number of measurements · Similar vectors = similar things · Word embeddings = words converted to vectors · Cosine similarity = a measure of how similar two vectors are

Matrices — Grids of Numbers That Transform Vectors

A matrix is a 2-dimensional array of numbers. In AI, matrices are the "weights" of a model — the learned parameters. When input text is converted to a vector, the model multiplies that vector by a series of matrices to produce the output. This is "matrix multiplication" — the core operation of every neural network.

The key intuition: a matrix transforms one vector into another vector. It changes the position of a point in space. The model's millions of matrices collectively transform "input text vector" into "appropriate output token probability vector." Every matrix multiplication is one step in that transformation.

The Dot Product — How Similarity is Measured

The dot product of two vectors is a single number that measures how similar they are in direction. High dot product: vectors point in the same direction — the items they represent are similar. Zero dot product: perpendicular — unrelated. Negative dot product: opposite directions — dissimilar.

In Phase 2, you will build a vector retrieval system (RAG). When a user asks a question, the system converts it to a vector and finds documents with the most similar vectors. The similarity is measured by dot product. Understanding this now means Phase 2 is comprehension, not memorisation.

Chapter Twenty-Four
Probability Foundations
What a language model is actually doing when it generates text

A language model does not know the answer to your question. It computes a probability distribution over all possible next tokens — and samples from that distribution. Understanding this is the difference between treating AI as magic and understanding it as a statistical system you can reason about, tune, and govern.

The Experienced Officer's Prediction Analogy
An experienced SBI officer who has processed 10,000 loan applications has an intuition: "Given this applicant's profile, there is a 75% chance they will be approved, 20% chance referred for additional documents, 5% chance rejected outright." They do not know with certainty — they assign probabilities based on patterns. A language model does the same: given the text so far, it assigns a probability to every word in its vocabulary as the next token. It then samples one word from that probability distribution.

The Softmax Function — Turning Numbers into Probabilities

The model's final layer produces raw numbers called logits — one number per word in the vocabulary (typically 32,000–100,000 words). Logits are not probabilities. Softmax converts them: it takes the exponential of each number and divides by the sum of all exponentials. The result is a probability distribution that sums to exactly 1.0.

Temperature — Controlling Randomness

Before softmax, the logits are divided by a number called temperature. Temperature = 1.0 is the default. Lower temperature (0.1) makes the distribution sharper — the highest-probability word becomes overwhelmingly likely, outputs become repetitive and deterministic. Higher temperature (1.5) makes the distribution flatter — more randomness, more creativity, more hallucination.

Why This Matters for Enterprise AI Governance
When an AI system in a regulated environment produces a factually incorrect statement (hallucination), it is not random malfunction — it is the model sampling from a probability distribution where the correct answer had lower probability than a plausible-sounding but incorrect answer. Fixing this requires: better prompting (raising correct answer's probability), lower temperature (reducing randomness), or HITL review (catching the output before it reaches the customer). These are governance decisions. Phase 4 of this curriculum is built on this understanding.

Exponential Backoff
The most important error handling pattern in enterprise integration
The Polite Visitor Analogy
A polite visitor who knocks on a door and gets no answer waits a moment, then knocks again, waits a little longer, knocks a final time, then leaves. They do not hammer the door repeatedly every second. Hammering repeatedly makes the problem worse — if the server is overloaded, flooding it with rapid-fire retries makes it more overloaded. Exponential backoff gives the server time to recover: wait 2 seconds, then 4, then 8. This is the standard pattern in every enterprise integration — SWIFT retry rules, payment gateway retry policies, and your API client all follow it.
Exponential Backoff — The Code Explained
for attempt in range(1, retries + 1):
Loop from 1 to retries (inclusive). range(1, 4) gives [1, 2, 3]. Starting at 1 (not 0) means attempt numbers are human-readable in log messages: "attempt 1/3", "attempt 2/3".
if attempt < retries: delay = RETRY_DELAY_SECONDS * (2 ** (attempt - 1)) time.sleep(delay)
2 ** (attempt - 1) — 2 to the power of (attempt minus 1).
Attempt 1: 2⁰ = 1 × 2 = 2 seconds.
Attempt 2: 2¹ = 2 × 2 = 4 seconds.
Attempt 3: 2² = 4 × 2 = 8 seconds.
time.sleep(delay) — pause execution for delay seconds.
The condition attempt < retries — don't sleep after the last attempt. No point waiting after you've decided to give up.
Mocking in Tests
Why tests use fake API responses — and how

Your test_api_client.py uses unittest.mock to replace the real API call with a fake response during testing. This is called mocking. Understanding why this is necessary and how it works is essential because mocking appears throughout Phases 2–4.

The CBS Training Environment Analogy
SBI staff train on a CBS test environment, not the production system. If training exercises used live CBS, every practice transaction would affect real accounts. Mocking in tests works identically: instead of calling the real exchange rate API (which might be slow, unavailable, or charge per call), the test replaces it with a controllable fake. The fake always returns a predetermined response, making tests fast, free, and repeatable.
Mocking — The @patch Decorator Explained
@patch('api_client.requests.get') def test_fetch_uses_correct_url(mock_get):
@patch('api_client.requests.get') — a decorator that replaces requests.get inside api_client with a fake object called mock_get for the duration of this test.
mock_get — the fake replacement. You control what it returns. In the real code, requests.get(url) makes an HTTP call. In the test, mock_get(url) returns whatever you configure it to return — instantly, without any network access.
mock_response = MagicMock() mock_response.json.return_value = MOCK_RESPONSE mock_response.raise_for_status.return_value = None mock_get.return_value = mock_response
Configuring the fake:
MagicMock() creates a flexible fake object that can simulate any method calls.
mock_response.json.return_value = MOCK_RESPONSE — when .json() is called, return the MOCK_RESPONSE dict.
mock_response.raise_for_status.return_value = None — when raise_for_status() is called, do nothing (no error).
mock_get.return_value = mock_response — when requests.get() is called, return this fake response object.
